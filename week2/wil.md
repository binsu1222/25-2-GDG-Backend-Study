## 스프링 계층형 아키텍처
- 브라우저(프론트엔드) : 손님
- 컨트롤러 : 브라우저와 소통, 하지만 비지니스 로직을 몰라서 Service에게 시킴
- 서비스 : 전체적인 레시피 (비지니스 로직)을 모두 알고 있음, 근데 재료가 없음
- DAO (Data Access Object): 창고 관리인
- DAO / Repository : 창고 어느 위치에 특정 재료가 위치했는지 알고 있음.
- DB : 냉장/냉동 창고

- DTO : 주문서, 영수증 (주문에 꼭 필요한 정보만 포함), 소통 목적에 맞는 필요한 정보만 전달하고자 함
- Entity : 원재료 = 모든 정보 포함 (DB 테이블과 맵핑되는 핵심 객체)

### 과정
1. Repository가 DB에서 특정 재료를 꺼내와서, 꺼낸 재료를 서비스 계층까지 보낸다.
2. 서비스 계층이 요리를 해서 (비지니스 로직), 컨트롤러에게 보낸다.
3. 최종적으로 컨트롤러가 브라우저(손님)에게 완성 요리를 전달한다.

### 컨트롤러
- HTTP 요청과 응답을 받는 역할
- 특정 endpoint로 요청을 가장 먼저 처리하고, DTO를 사용해서 서비스 계층과 데이터를 주고받음

## 패키지 구조
- 계층형 구조 : Controller, Service 등 같은 기능을 하는 애들끼리
- 도메인형 구조 : Member, Product 등 도메인 별로 구분해서 그 내부에 컨트롤러, 레퍼지토리 등이 공통적으로 존재

## 상태 코드(status code)
1. 200 OK (처리 성공)
2. 201 Created (데이터 생성 성공)
3. 204 No Content (요청은 성공인데 보내줄게 없음, 예를 들어 Delete 통한 영구 삭제)
4. 400 Bad Request (클라이언트 요청 오류)
5. 404 Not Found (요청 데이터 없음)
6. 500 Interval Server Error (서버 에러)
